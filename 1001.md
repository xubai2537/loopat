# 1001 —— AI 协同 Workspace

> 内部代号 **1001**。一句话：人 + AI 共享的、由 4 个 tab 构成的协同工作空间。
> MVP 标准：用一次觉得"哇，是这个东西"。

---

## 哲学：人类的三种稀缺资源 ↔ 三个 tab

> Driver 是人类。AI 没有自主欲望、没有真正的注意力、不会熵减 —— 这些只能从人来。

| 人类稀缺资源 | 对应 tab | 含义 |
|---|---|---|
| **欲望驱动力** | **Loop** | "我决定要做 X" —— 人类愿意把时间投进去的方向 |
| **注意力** | **Focus** | "现在我在意的是什么" —— 当下、有限、需要保护 |
| **熵减能力** | **Context** | "把混乱整理成清晰" —— 文档 / agent / repo 的精简，只有人参与才会越来越简 |

Chat tab 是**协调通道**（同步、共识形成），不属于稀缺资源轴。

设计含义：
- **Loop tab** 强调 driver / 主动开 loop / fork
- **Focus tab** 强调 zen、当下、自动归档（不堆东西，对抗注意力稀释）
- **Context tab** 强调精简、可信、人审过的 knowledge（不放弃熵减给 AI）

---

## 顶层：Workspace

10 人左右共享一个 **Workspace**。Workspace 是隔离边界 —— 跨 workspace 的东西不互通。

类比：Slack 的 workspace、Notion 的 workspace。

---

## 主 UI：4 个 Tab

### 1. Context Tab

> **Context = team's distilled materials.**
> 团队精炼过、可被反复使用的原料。三种形态都是同一种"熵减"活动的产物。

```
Context
├─ Knowledge  passive · markdown        ← 静态思考、人读
├─ Agents     active · executable       ← 静态思考 + 执行外壳
└─ Repos      passive · code            ← 静态代码、机器读
```

**关键洞察**：Agent 不是 Context 的反例，是 Context 的进化形态：

- Knowledge 的 prompt / persona 写在 markdown 里，给人读
- Agent 的 prompt / persona 封装在自己内部，给自己跑
- 两者都是"被 distill 过的、可被 Loop 调用的"，都需要持续精简

人类把"读一遍 doc 然后做某事"这套流程，编码成 agent —— 这就是把 passive context 升级成 active context。

#### Knowledge（passive · markdown）

- **像 Obsidian** —— markdown + wikilink + backlinks + frontmatter
- 多人共同维护、有 revision
- **内部按熵减程度分层**（都是子目录，不是单独 tab）：
  - `inbox/` —— 临时草稿、零散记录（高熵，未沉淀）
  - `knowledge/`、`loop/`、`ai-org/` 等稳定主题目录 —— 沉淀的稳态知识（低熵）
- 名字叫 Knowledge 是因为这个 tab 的**最终产物**是稳态知识；inbox 是过渡态

#### Agents（active · executable）

- **跑在任何地方**：云、本地、容器、cron、k8s
- **只通过 1001 API 接入**：
  - GET `/context` 拉知识（git 仓 pull）
  - GET `/focus` 看团队当下注意力
  - SUB `/chat/<channel>` 订阅 channel
  - POST `/chat/<channel>` 发言
  - POST `/loops` 主动创 loop
- **每个 agent 有自己的 persona**（system prompt + tool 集 + 触发模式）
- **Trigger 有三种**：mention（被 @）/ schedule（定时）/ event（外部事件）
- **来源**：从反复出现的 Loop pattern 里 distill 出来

**Loop ↔ Agent 关系**：

| | Loop | Agent |
|---|---|---|
| 谁的 | 一个 driver（人）的事 | 团队资产（无人 driver）|
| 时长 | 数小时数天，会 close | 长期运行 |
| 触发 | 人决定开 | 持续监听 / 定时 |
| 终点 | artifact → close | 没终点（除非下线）|
| 改 persona | 不（一次性的事） | 有（迭代 prompt、加 tool）|
| 数量 | 多（几十）| 少（几个，每个明确职责）|

**Loop 是火，Agent 是值班的炉子。**

#### Repos（passive · code）

- 团队 register 的 git 仓
- 每个 repo 显示 recent loops + spawn loop 入口
- 点 spawn → 后台 `git worktree add` → 创新 loop

#### 设计哲学：约定大于配置

> Context 管理的核心原则。

**配置 = 加熵；约定 = 减熵。** Context 既然是熵减能力的产物，本身就不应该靠"无穷多可配置项"来运作。

具体落地：

- **Knowledge 结构有约定**：`inbox/` 是临时草稿、`knowledge/` 是稳态知识、`loop/` 是模型相关、`ai-org/` 是哲学。**不让用户自定义层级**
- **文件命名有约定**：inbox 文件 `YYYY-MM-DD.md`、loop 名走 kebab-case
- **Agent persona 有约定**：标准 tools name（`chat.read` `chat.post` `loop.create` `focus.update`）、统一 trigger 三种（mention / schedule / event）
- **Frontmatter 有约定**：`title / tags / updated / driver` 几个固定字段
- **Repo metadata 有约定**：name / remote / branch，不接受自定义字段

**为什么这对 AI 系统尤其重要**：
- 配置爆炸 → AI 无法稳定消费 → 每次都要重新理解
- 约定固定 → AI 一次学会，跨 team 可移植 → agent 能在不同 team 复用
- 配置项是用户的负担；约定是设计者的负担

**人类的 entropy-reduction 工作**：把一类潜在配置项压缩成一条约定。Convention 越多，配置项越少，系统越简洁，AI 越好用。

跟"AI 没有自主熵减能力"配对：**人类要主动定约定，不能让用户每个角落都配**。

### 2. Chat Tab

- **像 Slack** —— 多 channel、单聊 + 群聊
- **有 agent-based 机器人**
- bot 能力：
  - 帮人总结到 `inbox/`（临时记录）
  - 沉淀东西到 `knowledge/`（稳态知识）
  - 把讨论升级成 focus / loop
  - 总结会议纪要

#### Chat vs Loop（关键区分）

> **Chat 是入口，Loop 是深加工。**

| | Chat | Loop |
|---|---|---|
| 时长 | 几条消息 done | 数小时 / 数天 |
| 上下文深度 | 浅（一问一答）| 深（dir + 持续 chat + artifact）|
| Agent 角色 | 一次性回答（信息查询、统计、调侃）| Driver 的持续助手 |
| 例子 | "xx 命中率为啥降了" | "上线 gateway" |

**升级路径**：在 chat 里聊着聊着发现"这事得专门搞" → 在那条消息上 `spawn loop` → 转到 Loop tab 接着干。

**回流路径**：Loop 进展 / 摘要可以发回 chat（让团队感知），不需要别人都来 attach。

这跟"agent 回流"原则一致：chat 是低成本入口，loop 是高强度生产，两者通过 agent 桥接。

### 3. Focus Tab（不是 Todo）

- **当下视角** —— 团队**现在**在意什么。**没有 "this week" 概念，只有"当下"**
- **侧栏极简**：只有"当下 / 归档"两个 view
- **顶部 Pinned**：手动钉住的核心事项，**永不归档**
- **中段 focus**：手动加进来的注意力项，**8 天无活动 → 自动归档**（鼓励放手）
- **底段 active loops not in any focus**：当前任意人 driving、但没绑定到 focus 的 loop，自动出现
- **m:n 关联**：一个 focus 关联多个 loop，一个 loop 也可以同时出现在多个 focus 下
- **活动定义**：focus 关联的任意 loop 有新一轮 chat → focus 时间刷新
- **对抗注意力稀释**：focus 是 view，不是全集，**不焦虑**
- 全 team 共享

### 4. Loop Tab

> 每个 loop = 一个 dir + 一段持续 chat + 多个 driver 协作。

- **像 Claude 桌面端会话列表** —— 每个 loop 是一个会话
- **3 列 IDE 风格**：左 loop 列表 / 中文件树 / 右 chat + terminal drawer
- **chat 不显示个人名字**，统一 `user` / `ai`，driver 切换时插一行系统标记
  （`driver-change: from 阿尔萨斯 to simpx · 14:32`）—— 强调"这是某个 driver 当下在跟 AI 协作"，不是群聊
- 列出**有权限看的所有 loop**
- 有一个"**driver 是我**"的视图（这部分跟 todo 看着重叠但不一样）

## Focus vs Loop 的本质区别（关键澄清）

> **Focus 管"团队当下的注意力"，Loop 管"具体在做什么和怎么做"。**

| | Focus | Loop |
|---|---|---|
| 关心 | **现在**团队在意什么 | 怎么做、谁在 drive、过程 |
| 视角 | attention 维度（消减） | execution 维度（具体）|
| 数量 | 少（被注意力约束）| 多（有什么做什么）|
| 关系 | 一个 loop 可以挂在 0 ~ N 个 focus 下 | 不在 Focus 的 loop 仍正常存在 |
| 自动归档 | 8d 无活动 → 归档（pinned 永不归档）| 不归档（mtime 自然衰减）|

**例子**：
- 一个新探索 loop 不一定是 focus，先打 loopctl 这种
- 一个上线 loop（gateway-launch）肯定是 focus，pin 住
- 闲置 8 天的 focus 自动归档，但 loop 还在

## Loop 的执行模型（最大纠结点）

Loop 在哪儿"跑"？三个候选，各有 trade-off：

### 选项 A：只允许云端 loop
- ✅ 共享非常方便（云端就在那）
- ❌ 灵活性受限（不能本地跑、离线、用本地工具链）

### 选项 B：host loop + save/load
- ✅ 灵活性高（本地随便跑）
- ❌ 共享很麻烦（要打包上传 / 解压恢复）

### 选项 C：host loop + 远程 C/S 架构共享 ⭐
- ✅ 灵活性高 + 共享方便（本地跑，但开 client 端口让别人 attach）
- ⚠️ 安全性问题（暴露端口 / 权限边界 / 账号管理）

> **这是当前最大未决问题。** 三个选项各有真实 trade-off，没有"显然对"的。

类比：选项 A 像 Google Docs；选项 B 像 git 工作流；选项 C 像 VSCode Live Share / tmux + ttyd。

## Agent 在系统里的角色

**Agent 不只是 chat 里的回应者**，更是**信息整理者** —— 信息的"凝水器"：

```
chat / loop 过程 ──→ agent 提炼 ──→ inbox / knowledge / focus
```

具体能力（增量发现）：
- 监听 chat / loop 内容
- 自动整理到 inbox（粗粒度）
- 沉淀到 knowledge（细粒度，需人确认）
- 维护 focus（哪些项目被反复提，自动建议 pin）
- 生成会议纪要、周报等 view

**整个 work flow 被这个系统接管** —— 这是它跟 Slack / Notion / Obsidian 等单一工具的根本不同：**人 + AI 的协同工具，不是给人用的工具加 AI 插件**。

## MVP 的"哇"在哪

不是 4 个 tab 都做完才有"哇"。"哇"在于：

- 进来看到 workspace + 4 个 tab，结构一目了然
- 创一个 loop，里面跟 AI 聊天，AI 自动把关键信息流回 inbox
- 切到 Knowledge tab 看到 AI 整理过的笔记
- 别人能看到这个 loop（或被拉进来）
- 切到 Focus 看到团队当下在意的事

**关键**：4 个 tab 是 ONE 系统的 4 个面，不是 4 个独立工具拼接。AI 在中间穿针引线。
