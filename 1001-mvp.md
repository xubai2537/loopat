## 1. 概念精确定义

### 1.1 Loop

**定义**：first-class 工作单元 = **context + ai + workdir**，绑定一个长程任务和一个 driver。

**逻辑构成**：
- **context** —— 从 Context tab 拉进来的原料：关联的 repo、相关的 knowledge 文档、可调用的 agents
- **ai** —— AI runtime；MVP 阶段就是 fork 后的 opencode
- **workdir** —— 实际干活的目录（通常是某个 repo 的 git worktree）

> chat 是 user ⇄ ai 在 workdir 上交互的副产物，**由 opencode session 持久化**，1001 不重复存。

**1001 在 opencode 之上加的语义**：
opencode 已经管 session（chat 历史）和 project（工作目录）。1001 的 loop 在这之上加：
- **driver** —— 谁正在 drive 这个 loop（强单人语义，可 transfer）
- **context 注入** —— loop 创建时绑定一组 knowledge / repo / agents，注入系统提示
- **focus 关联** —— loop 可挂在 0~N 个 focus 下
- **attach 协议** —— 队友远程 attach 我的 loop（多 client mirror）

**生命周期**：

```
open ──active──→ closed
            │
            ├─ forked     (开新分支 loop)
            └─ archived   (long-term 不动)
```

**关键约束**：
- 一个 loop 同一时刻只有一个 driver（人）
- driver 可被 transfer（产生一条 driver-change 系统消息）
- chat 不显示个人名字（统一 `user` / `ai`），driver 切换通过系统标记呈现
- close 后仍可读、可 fork，但默认不再 active

**操作**：`spawn` / `fork` / `transfer-driver` / `close` / `archive`

### 1.2 Focus

**定义**：团队当下"什么重要"的 view。是注意力的物化，不是 todo list。

**结构**：每个 focus item 是一个简短标题 + 关联的 loop 集合（m:n）。

**三档**：
| 档 | 行为 | 来源 |
|---|---|---|
| **pinned** | 永不归档 | 手动 pin |
| **focus** | 8 天无活动自动归档 | 手动 add |
| **archived** | 不显示在默认视图 | 自动归档或手动 |

**活动定义**：focus 关联的任意 loop 有新一轮 chat → focus 时间刷新。
**附加显示**：底段列出"任意人 driving 但未挂 focus 的 active loop"，自动出现，不需要 add。

**约束**：
- focus 不是任务，没有 done 状态。归档 ≠ 完成
- focus 不绑定 driver
- m:n loop 关联（一个 loop 可挂多个 focus，反之亦然）

### 1.3 Context

> team's distilled materials —— 团队精炼过、可被反复使用的原料。

三种形态：

#### Knowledge（passive · markdown）

- `inbox/`：临时草稿、零散记录（高熵）
- `knowledge/`、`loop/`、`ai-org/` 等稳定主题目录：沉淀的稳态知识（低熵）
- 写法：markdown + frontmatter（`title / tags / updated / driver`）+ wikilink `[[X]]` + 自动 backlinks
- 多人共同维护，有 git revision

#### Agents（active · executable）

- 跑在任何地方（云、本机、容器、cron、k8s），通过 1001 API 接入：
  - `GET /context` 拉知识
  - `GET /focus` 看团队当下注意力
  - `SUB /chat/<channel>` / `POST /chat/<channel>` 收发消息
  - `POST /loops` 主动创 loop
- persona = system prompt + tool 集 + 触发模式
- Trigger 三种：`mention` / `schedule` / `event`
- 来源：从反复出现的 Loop pattern 里 distill 出来 —— **agent 是把"读 doc 然后做事"这套流程编码进可执行外壳**

#### Repos（passive · code）

- 团队 register 的 git 仓
- 每个 repo 显示 recent loops + spawn loop 入口
- spawn loop = 后台 `git worktree add` → 创建新 loop 的 work dir

**Loop ↔ Agent 对照**：

| | Loop | Agent |
|---|---|---|
| 谁的 | 一个 driver（人）的事 | 团队资产（无人 driver） |
| 时长 | 数小时数天，会 close | 长期运行 |
| 触发 | 人决定开 | 持续监听 / 定时 |
| 终点 | artifact → close | 没终点（除非下线） |
| 改 persona | 不（一次性的事） | 有（迭代 prompt、加 tool） |
| 数量 | 多（几十） | 少（几个，每个明确职责） |

### 1.4 Chat

- 多 channel + 单聊 / 群聊
- bot 是一次性回应（信息查询、统计、提醒）
- 可在某条消息 `spawn loop`，把讨论升级成深加工
- loop 的进展可回流进 chat，让团队感知，不需要别人 attach

**Chat 与 Loop 是搭档，不是替代**：chat 是 sync 协调层，loop 是异步深加工层。spawn 是单向升级（chat → loop），回流是可见性 broadcast（loop → chat）。两者职责不重叠，所以也不需要做选择。

### 1.5 概念关系图

```
chat ──[spawn]──→ loop ──[沉淀]──→ knowledge / focus
                    │
                    ├──→ repo (git worktree)
                    │
                    └──→ agent (反复出现的 pattern → 长期值班)
```

### 1.6 顶层组织

10 人左右共享一个 **workspace**（名字待定 —— team 更直接但 workspace 强调"空间"，可跨 team 复用）。
Workspace 是隔离边界，跨 workspace 的 loop / context / focus 不互通。

## 2. 阶段性目标

> 前两个 phase 定义 3-6 个月内不会再变的目标方向。后两个是落地里程碑：0.1 单人 → 0.2 协作。

### Phase 1：高保真原型

**目标**：把 1001 想要的功能和样式都完整呈现在前端原型里 —— 不涉及实现。

**产出**：
- 说明文档（本文 + `1001-story.md`）
- 高保真前端原型（4 个一级概念的核心 UI 都能 demo）

**退出标准**：reviewer 看完前端原型能讲清楚 1001 是什么。

**当前状态**：进行中。详见 `PROTOTYPE-STATUS.md`。

### Phase 2：架构选型

**目标**：选定一套 1-2 个月内能实现的架构。**必须有妥协，不追完美又复杂。**

**候选**（这一步要真正比较）：
- **opencode**（已 fork 跑通 Phase 1）—— TS 全栈、Tauri 桌面、SSE 多 client
- **pi.dev** 等 —— 待评估
- 自建 —— 不在 1-2 个月窗口里

**产出**：架构文档 + 选型理由（接受妥协 X 而不是妥协 Y 的取舍）。

> Phase 1 + Phase 2 完成后，3-6 个月的目标固定下来，不再 churn。

### Phase 3：0.1 单人版（升级版 ccx）

**目标**：100% 代替我本地的 ccx，连续用一周不回退。

**范围（按概念）**：

| 概念 | 0.1 范围 |
|---|---|
| **Loop** | 完整（无 attach）。spawn / chat / close 单人即可；前端跟原型差不多 |
| **Context** | Knowledge + Repos。**显示能力必须有；编辑可后置** |
| **Focus** | 类 ccx todo，先只起"关联 loop"作用 |
| **Chat** | 单聊 chatbot（不支持 group），体现 chat-agent 与 loop 的区别就够 —— 可后置 |
| **Agent** | 0（不在 0.1） |

**关键约束**：c/s 架构从 0.1 就要有 —— 即使只有自己用，为 0.2 铺路，避免中途返工。

**退出标准**：连续用一周，不再回去用 ccx。

### Phase 4：0.2 多人协作

**目标**：在 0.1 的 c/s 架构之上加入"attach 别人 loop"的能力。

**范围**：
- attach 协议（多 client mirror）
- driver-transfer
- Chat 升级到 group（多人 channel）
- 权限边界初步划线

**退出标准**：两人能完成"一人 spawn loop → 另一人 attach 接手 → close" 的完整流程。

## 3. 设计细节

### 3.1 4 个一级概念的 UI 范围（Phase 1 高保真原型）

**Loop tab**（最完整）
- 3 列 IDE 风格：左 loop list / 中文件树 / 右 chat + terminal drawer
- chat 不显示个人名字，driver 切换插系统标记
- "driver 是我"的视图

**Focus tab**（zen）
- 侧栏极简：当下 / 归档
- 主区三段：📌 Pinned / focus（8d expires）/ active loops not in any focus

**Context tab**（最大）
- 顶部 horizontal sub-nav：Knowledge / Agents / Repos
- 每个 sub-tab 独占 sidebar+main
- Knowledge 子 tab 是 MVP 的重点：Obsidian-like 体验

**Chat tab**（轻量）
- Slack-like channels + DMs（0.1 阶段简化为单聊 chatbot）
- 每条消息可 `spawn loop`

### 3.2 关键架构选择

**约定大于配置**
- 目录结构、文件命名、frontmatter 字段、agent persona 字段、API 端点 —— 都是约定
- 用户不能配置层级、字段名、tool 集语义

**统一的 C/S 架构**
- 每个 loop 跑在某个 server 上 —— server 既可以是本机也可以是云端
- client 走同一套 attach 协议
- 共享 local server 的安全 trade-off（端口、权限、账户）是协作不可避免的妥协
- **0.1 即采用 c/s 架构**，即使只有自己用 —— 这是 Phase 3 → Phase 4 平滑过渡的前提

## 4. 待解（按优先级）

1. **Loop 共享的安全边界**：C/S attach 协议的权限模型、本机 server 暴露端口的安全 trade-off 还没想清楚
2. **Workspace 隔离 + 权限**：账户系统、谁能看哪个 loop / focus / knowledge
3. **Agent 提炼回流的触发机制**：被动监听 / 定时跑 / on-demand 触发
4. **Knowledge 的 wikilink + backlinks 持久化索引**：原型是 mock，要有真实的反向索引和增量维护
5. **Loop 的 dir 与 repo worktree 映射**：spawn loop 时 git worktree add 的策略，close / archive 时 worktree 怎么回收
